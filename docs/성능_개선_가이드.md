# 성능 개선 가이드 (CPU 온프레 환경)

CPU에서 LLM을 돌리면 **답변 생성**이 가장 느립니다. 아래는 영향이 큰 순서로 정리한 개선 방법입니다.

## 0. 한눈에 보기

| 구간 | 주요 원인 | 해결 |
|------|-----------|------|
| 답변 생성 | CPU LLM 추론 | **작은 모델** + keep_alive + num_predict 제한 + 스트리밍 |
| 인덱싱 | 청크 1건씩 저장 / 큰 임베딩 모델 | **배치 삽입(완료)** + 가벼운 다국어 임베딩 |
| 검색 | top_k 과다 | top_k 축소 |

## 1. 코드에 이미 반영된 개선

- **인덱싱 배치 삽입**: 청크를 한 번에 저장 (이전엔 1건씩 → 수 배 향상)
- **임베딩 모델 1회 로드**: 프로세스당 한 번만 로드해 재사용
- **Ollama 연결 재사용 + keep_alive**: 모델 재로딩 방지
- **num_ctx / num_predict 제한**: 컨텍스트·출력 토큰을 줄여 생성 시간 단축

이 설정들은 `.env`로 조정합니다.

## 2. 가장 큰 레버 — LLM 모델 교체 (직접 적용 필요)

CPU에서는 모델 크기가 속도를 좌우합니다. 작은 instruct 모델로 바꾸세요.

```bash
# 예: 3B 이하 모델
ollama pull qwen2.5:3b-instruct
# 또는 더 빠르게
ollama pull qwen2.5:1.5b-instruct
```

`.env`:

```env
OLLAMA_MODEL=qwen2.5:3b-instruct
OLLAMA_KEEP_ALIVE=30m
OLLAMA_NUM_CTX=4096
OLLAMA_NUM_PREDICT=512
```

> 7B → 3B로만 바꿔도 CPU에서 보통 2~4배 빨라집니다. 품질이 충분하면 1.5B까지 시도.

## 3. 답변 토큰·컨텍스트 줄이기

- `OLLAMA_NUM_PREDICT`를 256~512로 (긴 답변이 필요 없으면 더 작게)
- `OLLAMA_NUM_CTX`를 2048~4096로
- `RETRIEVAL_TOP_K`를 6 → 3~4로 (LLM에 들어가는 컨텍스트가 짧아짐)

## 4. 임베딩(인덱싱) 빠르게

한국어 문서면 영어 전용 `bge-small-en`보다 다국어 모델이 정확하고, 작은 모델이 빠릅니다.

```env
EMBEDDING_MODEL=intfloat/multilingual-e5-small
EMBEDDING_BATCH_SIZE=64
EMBEDDING_THREADS=0   # 0=코어 수 자동
```

> 모델을 바꾸면 임베딩 차원이 달라지므로 **기존 인덱스는 재생성**해야 합니다(과제별 `전체 인덱싱` 다시 실행 또는 `storage/chroma` 초기화).

## 5. 체감 속도 — 스트리밍 (선택, 추가 작업)

`num_predict`로 줄여도 사용자가 "다 생성될 때까지" 기다리는 구조입니다.
토큰을 즉시 흘려보내려면 SSE 스트리밍이 필요합니다(백엔드 `/chat/stream` + 프론트 수정).
원하면 별도로 추가해 드립니다.

## 6. 하드웨어/배치

- 가능하면 **GPU 1장**이 가장 확실한 해결책입니다(CPU 대비 10배+).
- 동시 사용자가 많으면 인덱싱은 **백그라운드 작업 큐**로 분리.

## 7. 빠른 점검 체크리스트

- [ ] `OLLAMA_MODEL`을 3B 이하로?
- [ ] `OLLAMA_KEEP_ALIVE` 설정으로 재로딩 제거?
- [ ] `NUM_PREDICT` / `NUM_CTX` 제한?
- [ ] `RETRIEVAL_TOP_K` 축소?
- [ ] 임베딩 다국어·소형 모델 + 인덱스 재생성?
- [ ] (가능하면) GPU?
